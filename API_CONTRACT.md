# API Contract

Initial MVP routes:

```text
GET /health
POST /v1/device-session
GET /v1/templates
POST /v1/stories
GET /v1/stories
GET /v1/stories/{story_id}
POST /v1/stories/{story_id}/turns
POST /v1/feedback
```

Local development base URL:

```text
http://127.0.0.1:8000
```

Optional deployed base URL example:

```text
https://api.example.com/storycat
```

Notes:
- A deployment can expose the same route shapes either at root or behind a path prefix such as `/storycat`.
- The public repository does not include any production domain, Cloudflare account identifier, or active KV namespace ID.
- `LLM_FAKE_MODE=true` is the safe default for local development and template deployments.
- Story create, turn, fetch, and list flows can persist story records through the `STORYCAT_STATE` KV binding when you configure your own namespace.

## Implemented routes

### GET /health

Response:

```json
{"status":"ok"}
```

This route is intentionally unversioned so load balancers, deployment probes, and local setup checks can call it before the versioned MVP API is complete.

Request:

```json
{
  "device_id": "11111111-1111-4111-8111-111111111111",
  "template_id": "xianxia_rise",
  "locale": "zh-Hans",

    "main_goal": "查清家族没落真相",
    "special_ability": "能听见灵气裂隙中的低语"
  },
  "tone": "热血、悬念、成长",
  "content_rating": "teen"
}
```

Response:

```json
{
  "story_id": "uuid",
  "title": "裂隙听灵者",
  "opening_narrative": "string",
  "current_state": {},
  "choices": [
    {"id": "choice_1", "label": "低头忍耐，先观察局势", "risk": "low"},
    {"id": "choice_2", "label": "当众反击，争取试炼机会", "risk": "medium"},
    {"id": "choice_3", "label": "私下寻找掌事长老", "risk": "medium"}
  ]
}
```

Error:

```json
{
  "error": {
    "code": "template_not_found",
    "message": "Story template was not found.",
    "details": {"template_id": "missing_template"}
  }
}
```

Provider failure error:

```json
{
  "error": {
    "code": "story_generation_unavailable",
    "message": "Story generation is temporarily unavailable.",
    "details": {"reason": "provider_unavailable"}
  }
}
```

Notes:
- Fake mode stores stories in memory.
- Opening data is deterministic by template ID.
- The route uses backend settings to choose the story creation path.
- `LLM_FAKE_MODE=true` keeps the deterministic fake response behavior.
- `LLM_FAKE_MODE=false` enters the backend provider-factory story creation path; automated tests override that factory with a local fake provider and do not call external providers.
- Provider configuration, provider generation, or invalid provider-output failures return `story_generation_unavailable` with a stable sanitized `reason`.
- Public story creation errors must not expose raw provider messages, raw provider response text, missing backend settings, provider names, or API keys.
- Successful provider-generated story openings are recorded in the in-memory LLM ledger with provider, model, `story_bible_generation` task type, token usage, status, latency, and fallback metadata.
- Invalid provider-generated story openings are recorded in the in-memory LLM ledger with `parse_failed` status and schema failure metadata before returning the sanitized public error.
- Provider-generated story openings can use an explicitly injected in-memory router for quality-tier `story_bible_generation`; router-selected provider/model/fallback metadata is reflected in the internal response and ledger entry.
- Story-opening router selection failures return `story_generation_unavailable` with `details.reason=no_available_model`, skip provider generation, and create no ledger entry.
- Provider-generated story openings can use an explicitly injected in-memory user/story quota policy; quota preflight failures return `story_generation_unavailable` with `details.reason=user_token_budget_exhausted` or `details.reason=story_token_budget_exhausted`, skip provider generation, and create no ledger entry.
- Successful or schema-invalid provider-generated story openings increment injected user/story quota usage from the initial provider ledger entry.
- The public route keeps router and quota injection disabled by default.
- Deterministic fake-mode story creation is not a provider call and does not record a provider ledger entry.

```json
{
  "error": {
    "code": "missing_user_text",
    "message": "Free-text turn requires user_text.",
    "details": {}

```

Notes:
- Fake mode supports `input_type=choice` and `input_type=free_text`.
- Fake-mode next-choice labels are deterministic and turn-aware in Phase 5; clients must rely on the stable choice object shape, IDs, and risk values instead of treating example labels as fixed copy.
- The route uses backend settings to choose the turn generation path.
- `LLM_FAKE_MODE=true` keeps the deterministic fake-mode response behavior.
- `LLM_FAKE_MODE=false` enters the backend provider-factory turn generation path; automated tests override that factory with a local fake provider and do not call external providers.
- Choice input requires a valid `choice_id` from the current choices.
- Free-text input requires non-empty `user_text`.
- Provider, router, quota, validation, or provider-turn failures return `turn_generation_unavailable` with a stable sanitized `reason`.
- Public turn generation errors must not expose raw provider messages, raw provider response text, missing backend settings, provider names, or API keys.
- In Phase 2, the backend validates this shape with the `StoryState` schema.
- In Phase 2 fake mode, clearly unsafe, copyrighted-IP, or impossible free-text actions return a story-friendly redirect in the normal response envelope with `warnings` such as `action_redirected:unsafe_action`, `action_redirected:copyright_ip`, or `action_redirected:impossible_action`.
- Redirected free-text actions do not advance `turn_count`, chapter progress, danger, or relationship state.
- Redirected turns are kept in `latest_turns` with `redirected: true` and `redirect_reason`.
- In Phase 2 fake mode, `xianxia_rise` completes chapter 1 on the sixth valid normal turn and returns `warnings: ["chapter_completed:1"]`.
- Chapter-complete turns set `state.flags.chapter_1_completed`, `state.flags.chapter_1_completed_at_turn`, `state.flags.last_completed_chapter_index`, advance `current_chapter_index` to 2, and reset `current_scene_index` to 1.
- In Phase 5 fake mode, later `xianxia_rise` chapters also roll over at a deterministic scene boundary, returning `chapter_completed:{chapter_index}` warnings while preserving the existing `chapter_progress` object shape.

## StoryState schema notes

`current_state` and turn `state` are expected to include:

- story identity: `story_id`, `locale`, `template_id`, `title`
- protagonist profile
- story bible summary: world rules, tone, forbidden moves, major factions, main characters
- plot plan: total chapters and chapter plans
- progress: `current_chapter_index`, `current_scene_index`, `turn_count`
- active goal, short summary, and long summary
- relationships, inventory, stats, flags, and `updated_at`

Phase 2 fake-mode turn updates are deterministic:

- choice turns update progress, `stats.danger`, choice flags, summaries, and `relationships.npc_001`
- free-text turns update progress, `stats.danger`, free-text flags, summaries, and `relationships.npc_001`
- redirected free-text actions keep the current story state unchanged and return current choices
- chapter-complete turns can advance `current_chapter_index`, reset `current_scene_index`, and add chapter-complete flags

Phase 2 turn result state patches are validated before they can be applied:

- accepted patch fields: `active_goal`, `short_summary_append`, `relationships`, `inventory_add`, `inventory_remove_ids`, `stats_delta`, `flags_set`, and `chapter_progress_delta`
- relationship patches accept `affinity_delta`, `trust_delta`, and optional `status`
- stat deltas accept `danger`, `reputation`, `power`, and `health`
- malformed inventory items, malformed relationship/stat deltas, and unknown patch fields are rejected by the state manager validation helper
- fake-mode API response behavior is unchanged by patch validation

## Internal LLM provider contract

Phase 3 introduces an internal provider contract, not a public API route.

Supported task types:

- `story_bible_generation`
- `chapter_outline_generation`
- `normal_turn_generation`
- `state_extraction`
- `summary_generation`
- `json_repair`
- `safety_classification`
- `ending_generation`

Provider responses include:

- `provider`
- `model`
- `task_type`
- parsed JSON `content`
- raw JSON text
- usage metadata: `input_tokens`, `output_tokens`, and whether usage is estimated
- `latency_ms`
- `fallback_used`

The deterministic fake provider returns this contract without requiring an API key or network access.

OpenAI-compatible provider skeleton behavior:

- backend settings support provider name, base URL, API key, fast model, quality model, timeout, and fake-mode guard
- fake mode remains enabled by default and disables the OpenAI-compatible provider
- missing real-provider settings raise a typed `provider_not_configured` failure before any network call
- configured provider metadata includes provider, base URL, fast model, quality model, timeout, and fake-mode guard, but never includes the API key
- configured provider metadata includes a non-secret `transport_configured` boolean
- local Chat Completions payload construction includes selected model, ordered messages, JSON response format, and max output tokens
- local Chat Completions payload construction never includes the API key and does not send a request
- local/fake Chat Completions response envelopes can be parsed into the internal `LLMResponse` contract
- parsed envelopes preserve raw JSON text, parsed JSON content, usage metadata, latency, provider, model, task type, and fallback metadata
- missing provider usage is estimated from the request and response text so accounting fields remain populated
- malformed response envelopes raise typed `malformed_provider_response` failures
- calling `generate` without a configured transport raises typed `provider_unavailable`
- calling `generate` with an injected local transport builds the payload, calls that transport, and parses the returned local/fake envelope
- injected transport failures are wrapped as typed `provider_unavailable`
- an explicitly instantiated HTTP transport posts JSON to `{base_url}/chat/completions`
- HTTP transport sends the API key only as `Authorization: Bearer <key>` from the backend side
- HTTP transport never adds the API key to the Chat Completions JSON payload
- HTTP status failures are mapped to typed `provider_unavailable` failures
- malformed or non-object JSON response bodies are mapped to typed `malformed_provider_response` failures
- no default HTTP transport is installed automatically, so tests and fake mode still do not call Qwen or any external endpoint
- with mocked HTTP transport, the OpenAI-compatible provider can pass a valid normal-turn response through the internal gateway and ledger
- with mocked HTTP transport, invalid provider raw text still follows the existing `json_repair` and deterministic local fallback behavior
- settings-based OpenAI-compatible provider construction wires HTTP transport only when fake mode is disabled and provider settings are complete
- settings-based OpenAI-compatible provider construction does not call the provider or open network access

Internal app-level provider selection behavior:

- `build_llm_provider_from_settings` returns `FakeLLMProvider` when `LLM_FAKE_MODE=true`
- fake-mode provider selection does not require OpenAI-compatible base URL, API key, or model config
- `build_llm_provider_from_settings` returns the settings-wired OpenAI-compatible provider when `LLM_FAKE_MODE=false`
- real-provider selection preserves OpenAI-compatible typed configuration errors such as `provider_not_configured`
- provider selection constructs providers only; it does not call `generate` and does not open network access

Internal story-opening helper behavior:

- story-opening request construction uses `story_bible_generation`
- request metadata is built from `CreateStoryRequest` and the selected `StoryTemplate`
- request metadata includes locale, content rating, tone, template ID/name/genre/description/tags/recommended tone, and protagonist profile fields
- story-opening generation accepts an injected provider and validates the returned payload before route wiring
- valid story-opening payloads include `title`, `opening_narrative`, `story_bible`, `plot_plan`, `initial_state_patch`, and exactly three `choices`
- malformed story-opening payloads are rejected by internal schema validation
- validated story-opening payloads can be assembled into the current `StoryState` shape with story ID, story creation input, selected template metadata, initialized progress, summaries, relationships, inventory, stats, flags, turn count, and timestamp
- assembled story-opening state is validated through the existing state manager before it is returned
- the story service has an internal injected-provider helper that can generate, assemble, and store a provider-created `StoryRecord`
- the internal injected-provider helper returns `None` for unknown templates and does not write a record
- the story service has a settings-gated dispatcher that keeps deterministic fake creation when `LLM_FAKE_MODE=true`
- when `LLM_FAKE_MODE=false`, the dispatcher builds the provider through an injected/provider-factory boundary and then uses the internal LLM story creation helper
- unknown templates return `None` before provider construction in the settings-gated dispatcher
- `POST /v1/stories` now calls the settings-gated dispatcher; fake mode remains the default public behavior
- public story creation maps known provider and story-opening validation failures to sanitized `story_generation_unavailable` errors
- successful provider-generated story openings can record an initial success entry in the in-memory LLM ledger

Internal parser behavior:

- provider `raw_text` is parsed as JSON without throwing `JSONDecodeError` to callers
- non-object JSON is treated as invalid provider output
- `normal_turn_generation` and `json_repair` outputs are validated for `narrative`, exactly three `choices`, `state_patch`, `memory_update`, and `safety`
- parse failures return typed error metadata with `invalid_json` or `invalid_schema`
- normal turn generation can retry once with a `json_repair` task after parser failure
- repair retry results record `repair_used`, the initial parse result, repair response, repair parse result, and the final parse result
- if both initial parsing and repair fail, the internal gateway returns a deterministic local `normal_turn_generation` fallback payload with `fallback_used: true`
- fallback results preserve the original parse failure and repair parse failure metadata for later logging

Internal in-memory ledger behavior:

- the normal-turn gateway records to a default process-local in-memory ledger
- callers can pass an explicit in-memory ledger when tests or future services need isolated records
- successful initial provider attempts are recorded with status `success`
- failed initial provider attempts and failed repair attempts are recorded with status `parse_failed`
- deterministic local fallback events are recorded with status `fallback`
- each ledger entry includes provider, model, task type, attempt type, input tokens, output tokens, total tokens, estimated-usage flag, latency, fallback flag, and optional parse error metadata
- the ledger is local process memory only and is not exposed through a public API route

Internal in-memory router selection behavior:

- the router maps task type to `fast` or `quality` model tier using the internal task-tier mapping
- fake-mode model configs include provider, model, tier, priority, enabled flag, healthy flag, daily/monthly token budgets, token usage counters, and max output cap
- model selection checks configs in priority order within the required tier
- disabled, unhealthy, daily-budget-exhausted, and monthly-budget-exhausted models are skipped
- the selected model result includes provider, model, tier, max output cap, estimated input tokens, estimated total tokens, fallback flag, fallback chain, and skipped model metadata
- when no model is available in the required tier, router selection raises a typed internal failure that includes tier, fallback chain, skipped model metadata, and `no_available_model`
- router usage accounting can apply completed ledger entries to matching model configs
- non-fallback provider ledger entries increment matching model daily/monthly token usage by `total_tokens`
- local fallback ledger entries are ignored for provider model budget accounting
- normal-turn generation can receive an explicit in-memory router for the initial provider attempt
- when a router is provided, the initial normal-turn response and ledger entry use router-selected provider, model, output cap, and fallback metadata
- when router selection fails, normal-turn generation returns `ok: false`, `error_code: router_selection_failed`, and the typed router selection failure without calling the provider
- router-aware normal-turn generation records selected model usage from the initial provider ledger entry
- router-aware normal-turn generation returns an internal `router_usage_update` result for later logging or assertions
- deterministic local fallback entries are not counted by the gateway as provider model usage
- when no router is provided, existing fake provider metadata is preserved
- this is internal process-local routing only and does not call a real provider yet

Internal in-memory user/story quota behavior:

- callers can pass an explicit in-memory quota policy to normal-turn generation
- the quota policy can include user monthly token budget state and story monthly token budget state
- quota preflight estimates request tokens from prompt text plus the effective max output cap before the provider call
- if user quota is exhausted, normal-turn generation returns `ok: false`, `error_code: quota_exceeded`, and a typed `user_token_budget_exhausted` quota failure without calling the provider
- if story quota is exhausted, normal-turn generation returns `ok: false`, `error_code: quota_exceeded`, and a typed `story_token_budget_exhausted` quota failure without calling the provider
- quota preflight failures do not create fake ledger entries
- successful normal-turn provider attempts increment explicit user/story quota usage from the initial provider ledger entry
- quota usage updates include subject, subject ID, tokens added, resulting monthly usage, and remaining tokens
- deterministic local fallback entries are not counted as user/story quota usage
- when no explicit quota policy is provided, existing fake provider behavior is preserved

Internal provider-backed choice-turn helper behavior:

- the helper is service-internal and is not a public API contract change yet
- the helper builds a `normal_turn_generation` request from stored story state, selected choice, available choices, and choice metadata
- invalid choice IDs return `None` before provider generation and create no ledger entry
- successful provider choice turns apply the validated generated `state_patch` through the state manager
- generated choice turns replace the current choices with the provider output choices and append generated turn metadata to the story record
- generated turn metadata includes state patch, memory update, safety metadata, provider, model, token usage, repair flag, fallback flag, and warnings
